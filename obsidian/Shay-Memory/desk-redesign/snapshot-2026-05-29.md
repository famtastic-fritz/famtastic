## Executive snapshot

- Shay Desktop is a real, wired Electron 39 + React 19 app (~10.7k LOC main, ~10.7k LOC renderer, 30 main modules, 41 renderer source files, 128 IPC channels) with no TODO/FIXME debt and every screen wired to a live `window.hermesAPI.*` backend — not a mocked shell.
- The renderer covers 21 screens (Chat, Sessions, Agents, Studio, Office, Kanban, Models, Providers, Skills, Soul, Memory, Tools, Schedules, Gateway, Settings, plus the App/Welcome/Install/Setup/Splash/Layout chrome) but most of them are gated off in pure remote-HTTP mode via `RemoteNotice` placeholders.
- Coverage gap vs. Shay CLI is enormous: the CLI exposes ~200+ subcommands across `auth`, `fallback`, `webhook`, `pairing`, `mcp` (live add/test/configure/login/serve), `acp`, `checkpoints`, `curator`, `insights`, `dashboard`, `logs`, `hooks`, `claw`, `debug`, `backup/import`, `doctor`, `dump`, `update`, `uninstall`, `completion`, `computer-use`, and a dynamic plugin-supplied command surface — almost none of which have first-class Desktop UI.
- Architecture is leaner than expected: chat is plain HTTP+SSE to `127.0.0.1:8642/v1/chat/completions`, sessions are read by opening Shay's `state.db` directly with `better-sqlite3` (schema coupling risk), MCP servers are surfaced via a regex parse of `config.yaml` (read-only), there is no Keychain/safeStorage/OAuth, and the renderer has no Redux/Zustand — only React Context + lifted `useState`.
- First-build priorities: (a) a shared modal/dialog primitive + unified top-bar/status-bar, (b) first-class MCP add/test/configure UI, (c) Auth screen (pooled creds, fallback chain, Spotify, Nous OAuth) backed by `safeStorage`, (d) Logs/Diagnostics/Doctor/Dump/Debug-share surfaces, (e) Background-tasks + Notifications center matching Claude Desktop baseline.

- **Top bar / app chrome.** No dedicated top bar component. The mac traffic-light area is just `<div className="drag-region" />` injected by `App.tsx`; per-screen headers live inside each screen. For Chat, the header is `screens/Chat/ChatHeader.tsx` (title + token/cost `UsageBadge` + fast-mode toggle + new-chat + clear). Other screens render their own h1/h2 inline.
- **Sidebar.** `screens/Layout/Layout.tsx` — a static `NAV_ITEMS` array of 15 entries (chat, sessions, agents, studio, office, kanban, models, providers, skills, soul, memory, tools, schedules, gateway, settings) mapped to `lucide-react` icons via `assets/icons/index.tsx`. Active item gets `.active`. Footer shows the auto-update button (driven by IPC events) and the active profile label.
- **Chat window.** `screens/Chat/Chat.tsx` — composes `ChatHeader` → `.chat-messages` (scroll container managed by `hooks/useChatScroll`) → either `ChatEmptyState` (six suggestion buttons) or `MessageList` → `.chat-input-area` (`ChatInput` + `ModelPicker`). Drag-and-drop overlay (`.chat-drop-overlay`) sits over the whole container.
- **Message renderer.**
  - `screens/Chat/MessageList.tsx` — memoized list of `MessageRow` + a `TypingIndicator` that swaps in `toolProgress` text when present.
  - `screens/Chat/MessageRow.tsx` — avatar (`HermesAvatar` for agent, "U" tile for user) + `.chat-bubble`. Agent content runs through `components/AgentMarkdown.tsx`; user content is plain text. ### What's wired vs stubbed

Effectively every screen is wired to real backend data via `window.hermesAPI.*` — none of the screens use mock arrays, fixture JSON, or simulated responses. The only "stub" surfaces are intentional:

- **`RemoteNotice` for every remote-mode-gated feature.** In remote (HTTP) mode the following screens are replaced wholesale by a "not available in remote mode" placeholder, because the data lives server-side: Sessions, Agents (Profiles), Studio, Providers, Skills, Soul, Memory, Tools, Kanban, Gateway. The Chat, Models, Schedules, Office, and Settings screens stay live in remote mode. SSH-tunnel mode has full access — only pure HTTP remote restricts (`Layout.tsx:97`).
- **`SplashScreen` `onFinished`** is a no-op (`App.tsx:103-104`); transition is driven by the install check completing, not a timer. Documented in-line.
- **`Tools.tsx`** includes a `todo` icon key (line 218) — that is a toolset called "todo", not a code TODO.
- No `TODO` / `FIXME` / `HACK` comments anywhere under `src/renderer/src/`. Every `placeholder` hit is a literal input-placeholder string.

### Notable gaps

- **No shared modal/dialog primitive.** Every screen reimplements overlays. Image preview, drop overlay, slash menu, confirm-delete, profile-create, edit-task, edit-cron, edit-model all roll their own markup and outside-click handling. Easy target for a `<Modal>` extraction.
- **No top-bar component.** App chrome is just the mac traffic-light `.drag-region`; per-screen titles/headers are bespoke, so cross-screen actions (search, profile switcher, global status) have no home. The chat-only `ChatHeader` is the only formal header.
- **No right panel.** Layout is two-column. There is no inspector/details rail — the Studio and Office "details" appear as slide-down settings/logs panels inside each screen, and the Kanban task editor opens as a modal.
- **No router.** `Layout.tsx` is a hand-rolled `display:none` tab switcher (`paneStyle`, `visitedViews`). No deep linking, no back/forward, no per-screen URL — works but blocks browser-style navigation and `useParams` patterns. Cmd+N / Cmd+K are handled via menu IPC, not a keymap layer.
- **Single-screen "owned by parent" state.** `messages` and `currentSessionId` are lifted into `Layout.tsx` so the Chat pane survives tab switches, but everything else (Office state, Studio state, Kanban tasks) is owned inside its screen and refetched on visit, then cached via `visitedViews`. Inconsistent.
- **Approval detection is regex-based.** `MessageRow.tsx` uses `APPROVAL_RE` over message content to decide whether to render the Approve/Deny bar. Fragile — phrasing changes from the backend will silently hide the buttons. No structured "tool call awaiting approval" message type in `ChatMessage`.
- **`ChatMessage` doesn't carry tool-call metadata.** `MessageList`/`MessageRow` show a flat user/agent bubble; tool progress lives in a separate `toolProgress: string | null` channel rendered as a footer string. There is no first-class "tool call" / "tool result" / "thinking" message kind in the renderer.
- **Hardcoded suggestion list.** `ChatEmptyState.tsx` has six hardcoded `SUGGESTIONS`; not driven by user history or profile.
- **Mega-Settings file.** `Settings.tsx` carries 25 IPC calls and the bulk of the remote/SSH config UX, duplicating the panel logic that already exists in `Welcome.tsx`. Ripe for splitting into sub-sections.
- **Status surfaces are scattered.** Update state lives in sidebar footer; usage badge lives in `ChatHeader`; gateway running pill lives in Gateway screen; verify-warning lives at content-top. No unified status bar.
- **No "right panel" or thread/context view in Chat.** Sessions live on their own screen rather than as a collapsible history rail next to the conversation — switching sessions means leaving Chat.

## 4. Architecture (Desk ↔ Gateway)

### 4.1 Wire protocol — direct local HTTP with SSE streaming

Desk talks to the Shay backend over plain HTTP at `http://127.0.0.1:8642`, the default Shay gateway port. There is **no WebSocket, no streamable‑HTTP MCP, and no IPC to a sidecar helper** for the chat channel — the renderer routes through Electron IPC to the main process, and the main process uses Node's `http.request` to POST to the gateway's OpenAI-compatible Chat Completions endpoint. The response is consumed as Server-Sent Events.

- Base URL constant: `~/famtastic/shay-desktop-electron/src/main/hermes.ts:29` — `const LOCAL_API_URL = "http://127.0.0.1:8642"`.
- URL resolver (local / remote / SSH tunnel): `src/main/hermes.ts:31` (`getApiUrl()`).
- Chat request site: `src/main/hermes.ts:432-512` — `chatUrl = ${getApiUrl()}/v1/chat/completions`, POSTed via `http.request` (or `https.request` for remote HTTPS), with `stream: true` and an `Authorization: Bearer <API_SERVER_KEY>` header (only set when in remote/SSH mode — see `getRemoteAuthHeader()` at `hermes.ts:61`).
- SSE parser: `src/main/hermes.ts:466-501` — splits the body on `\n\n`, parses `event:` and `data:` lines, dispatches `delta.content` to `onChunk` callbacks, watches for `[DONE]`, and reads the final `usage` block (tokens + cost). Tool progress lines arrive as either custom SSE events or inlined `🔍 tool_name` strings in the content delta (legacy fallback at line 417). Custom event types include `hermes.tool.progress`.
- Auxiliary SSE-parser helpers live at `src/main/sse-parser.ts`.

Gateway side — the same endpoint:

- Server: `~/famtastic/shay-shay/gateway/platforms/api_server.py` (aiohttp-based, not FastAPI).
- Port constants: `api_server.py:57-58` — `DEFAULT_HOST = "127.0.0.1"`, `DEFAULT_PORT = 8642`.
- Route registration: `api_server.py:3338-3361` — `/health`, `/health/detailed`, `/v1/health`, `/v1/models`, `/v1/capabilities`, `/v1/chat/completions`, `/v1/responses`, `/v1/responses/{id}` (GET/DELETE), `/api/jobs` family, and the `/v1/runs` family (with SSE event stream at `/v1/runs/{run_id}/events`).
- Endpoint handler: `_handle_chat_completions` at `api_server.py:971`. Streaming runs through `web.StreamResponse` and emits OpenAI-compatible SSE deltas plus custom `event:` blocks.
- Session continuity is opt-in via the `X-Shay-Shay-Session-Id` request header (documented at `api_server.py:5`); the gateway echoes the session id back in the `X-Hermes-Session-Id` response header, which Desk reads at `hermes.ts:443`.

Gateway lifecycle is managed by Desk itself: `startGateway()` in `src/main/hermes.ts:861-905` spawns the Python `shay gateway` CLI as a detached child with `API_SERVER_ENABLED=true`. Health is polled via `GET /health` (`apiServerAvailable` flag). IPC handles: `start-gateway`, `stop-gateway`, `gateway-status` in `src/main/index.ts:731-748`. The renderer surfaces this in the **Gateway** screen (`src/renderer/src/screens/Gateway/Gateway.tsx`).

### 4.2 MCP servers / tools in the Desk UI

MCP servers are not negotiated at runtime over MCP protocol from inside Desk — Desk **reads them out of the Shay config file** and presents a read-only inventory plus an enable/disable toggle for the built-in toolsets.

- Renderer screen: `src/renderer/src/screens/Tools/Tools.tsx`. Two grids:
  - Built-in toolsets (`web`, `browser`, `terminal`, `file`, `code_execution`, `vision`, `image_gen`, `tts`, `skills`, `memory`, `session_search`, `clarify`, `delegation`, …) — defined in `src/main/tools.ts:14-95`. Toggled by writing the `platform_toolsets.cli` list in `~/.shay/config.yaml` (read at `tools.ts:170`, written at `tools.ts:193`).
  - MCP servers — defined in `Tools.tsx:252-258` (`McpServer { name, type, enabled, detail }`) and rendered at `Tools.tsx:334-341`.
- MCP source: `listMcpServers()` in `src/main/installer.ts:1186-1242` does a hand-rolled regex parse of the `mcp_servers:` block in `~/.shay/<profile>/config.yaml`, marking each server as `http` (has `url:`) or `stdio` (has `command:`) and pulling the `enabled` flag.
- IPC: `list-mcp-servers` registered at `src/main/index.ts:1241-1242`; preload binding at `src/preload/index.ts:825-829`.

Actual MCP tool invocation happens server-side inside Shay/Hermes — Desk never speaks JSON-RPC directly to an MCP server. The renderer only sees tool progress strings flowing back via the SSE stream (the `hermes.tool.progress` custom event and the legacy inline pattern in `hermes.ts:418`).

### 4.3 Sessions — created, restored, listed (NOT through the gateway)

This is the most surprising part of the architecture: **session listing and history are not fetched from the gateway over HTTP**. The Desk main process opens the Shay state SQLite database directly, read-only.

- DB path: `src/main/sessions.ts:8` — `const DB_PATH = join(HERMES_HOME, "state.db")` (i.e. `~/.shay/state.db`).
- Direct sqlite reader: `getDb()` at `sessions.ts:121-124` using `better-sqlite3` in `readonly: true` mode.
- Listing: `listSessions(limit, offset)` at `sessions.ts:126-169` runs `SELECT … FROM sessions ORDER BY started_at DESC`. The schema is owned by the Hermes agent; multimodal content uses a `\x00json:` sentinel prefix in `messages.content` (see `decodeContent()` at `sessions.ts:46-94`) — Desk reverse-engineered that format.
- History per session: `get-session-messages` (IPC at `src/main/index.ts:786`); `delete-session` (IPC at `index.ts:793`) deletes rows from both `messages` and `sessions` tables locally.
- FTS search: `search-sessions` (IPC at `index.ts:959`) queries the `messages_fts` virtual table when present (`sessions.ts:177-234`).
- Session **resume** in chat: the renderer passes the previously seen session id as `resumeSessionId` in the `send-message` IPC call (`src/renderer/src/screens/Chat/hooks/useChatActions.ts:75`). The main process forwards it as `session_id` in the POST body (`hermes.ts:290`); the gateway echoes it back via the `X-Hermes-Session-Id` response header on the first chunk (`hermes.ts:443`). New sessions: gateway mints an id on first turn and Desk captures it.
- Sessions list/search screen: `src/renderer/src/screens/Sessions/Sessions.tsx:168-201` calls `listCachedSessions`, `syncSessionCache`, `searchSessions` via `window.hermesAPI`.

### 4.4 Auth — Bearer token only, plain JSON, **no Keychain / safeStorage / OAuth**

Despite the Settings UI placeholders mentioning "Bearer token (API_SERVER_KEY)", the auth model is just a shared secret kept on disk in cleartext.

- Connection config struct: `src/main/config.ts:18-23` — `ConnectionConfig { mode: "local"|"remote"|"ssh", remoteUrl, apiKey, ssh }`.
- Store location: `src/main/config.ts:34-36` — `~/.shay/desktop.json` (literally `JSON.stringify(data, null, 2)`, no encryption — see `writeDesktopConfig()` at `config.ts:48-53`).
- Auth header builder: `src/main/hermes.ts:61-72` (`getRemoteAuthHeader`). In `local` mode no `Authorization` header is sent at all — the gateway only enforces a key when one is set in env (`API_SERVER_KEY`) AND the listener is non-loopback (`api_server.py:3372-3382`, `_authenticate` at `api_server.py:691-707` using `hmac.compare_digest`).
- Credential pool (provider API keys, e.g. Anthropic, OpenAI, Bedrock keys passed through to the gateway env on spawn): `src/main/config.ts:704-747`, stored in `~/.shay/auth.json`. Same story — plain JSON.
- **No `electron.safeStorage` import. No `keytar` dependency.** Verified by grep across `src/` and `package.json`.
- **No OAuth / setup-token flow.** There is no `setup-token`, no PKCE handler, no `claude.ai/setup-token` redirect, no browser-callback handler in the codebase. The Welcome / Setup screens (`src/renderer/src/screens/Welcome/Welcome.tsx`, `screens/Setup/Setup.tsx`) only collect a mode (local/remote/ssh), a URL, and an optional Bearer token; the i18n strings calling that token an "API_SERVER_KEY" make the model explicit.
- Remote/SSH variants: `src/main/ssh-tunnel.ts` opens a local-forward SSH tunnel from `localPort: 18642 → remote:8642`; `src/main/ssh-remote.ts:1119` reads the remote `.env`'s `API_SERVER_KEY` and feeds it to `setSshRemoteApiKey()` (`hermes.ts:55-59`) for subsequent requests.

### 4.5 Renderer state management — React Context + local `useState` only

There is **no Redux, Zustand, Jotai, Recoil, or MobX** anywhere in the renderer or in `package.json`.

- Global state is held in two React Contexts:
  - `ThemeProvider` at `src/renderer/src/components/ThemeProvider.tsx:1-78` (light/dark).
  - `I18nProvider` at `src/renderer/src/components/I18nProvider.tsx` + `I18nContext.ts`.
- All other state — current screen, chat messages, sessions list, tools/MCP inventory, loading flags, abort handles — is plain component-local `useState` lifted to the nearest common ancestor (typically `App.tsx` at `src/renderer/src/App.tsx:17-27`, `Layout.tsx`, or each screen module). Cross-screen events from the main process come in as raw IPC listeners (`window.hermesAPI.onChatChunk`, `onChatDone`, etc.) registered in the relevant `screens/Chat/hooks/*` files.
- Chat actions custom hook: `src/renderer/src/screens/Chat/hooks/useChatActions.ts` is the closest thing to a "store" — it threads `messages`, `setMessages`, `isLoading`, `setIsLoading`, `hermesSessionId`, and `profile` through callbacks.

### 4.6 Shared types

There is no single `shared/types.ts` and nothing is generated from the Python side. Type sharing is split:

- `src/shared/` holds shared **value-and-type** modules used by main + preload + renderer:
  - `src/shared/attachments.ts` — `Attachment`, `AttachmentKind`, `isImageMime`, `escapeXmlAttr` (the most-shared module; imported by `src/main/hermes.ts:24-27`, `src/main/sessions.ts:5-6`, `src/renderer/src/screens/Chat/types.ts:1`, `src/renderer/src/screens/Chat/attachmentUtils.ts:8`, `src/renderer/src/screens/Chat/ChatInput.tsx:21`).
  - `src/shared/askpass.ts` — sudo askpass IPC payloads.
  - `src/shared/i18n/` — `AppLocale`, message dictionaries.
- The **IPC API contract** is defined as a TypeScript interface in `src/preload/index.d.ts:114` (`interface HermesAPI`) — 851 lines of method signatures attached to `window.hermesAPI`. The implementation that binds those methods to `ipcRenderer.invoke(...)` is `src/preload/index.ts:175-829`. This file is the de-facto "API schema" between renderer and main.
- Domain types (sessions, kanban tasks, install progress, model config) are declared **inline** in `src/preload/index.d.ts:15-112` (e.g. `InstallStatus`, `KanbanTask`, `KanbanBoard`, `KanbanComment`, `KanbanEvent`, `KanbanRun`, `KanbanTaskDetail`, `KanbanCreateTaskInput`) and the renderer imports them via `window.hermesAPI`'s return types. Some screen-local types (`SessionSummary`, `SearchResult`) are re-declared in `src/main/sessions.ts:15-24,111-119` and not re-exported to the renderer — the renderer pulls them in by structural typing through `window.hermesAPI.listCachedSessions()`'s declared return shape.
- Nothing is generated from the gateway's Python schemas; the OpenAI-compatible request/response shapes are hand-written inline in `src/main/hermes.ts`.

### Notable findings worth flagging

1. **Sessions architecture coupling.** Desk reads Shay's `state.db` schema directly with `better-sqlite3`, including the `\x00json:` sentinel format for multimodal content (`sessions.ts:13,46`). Any future Shay-side migration of that table or sentinel format will silently break Desk's session list/history/search.
2. **No Keychain / safeStorage.** API keys, provider keys, and remote `API_SERVER_KEY` all live in cleartext under `~/.shay/desktop.json` and `~/.shay/auth.json`. The UI calling these "Bearer tokens" suggests stronger security than is actually present.
3. **`API_SERVER_KEY` is only enforced when the gateway binds non-loopback** (`api_server.py:3372-3382`); the default `127.0.0.1` deployment is fully unauthenticated. Anything on the local machine can hit `:8642`.
4. **No OAuth / setup-token flow at all** — the system has no concept of provider OAuth; everything is shared-secret API keys passed into the gateway as env vars when Desk spawns it (`hermes.ts:874-880`).
5. **No state library.** Chat, sessions, tools, settings — all are local `useState` lifted to App/Layout. As Desk grows this is the single largest architectural risk for renderer maintainability.

## Notable cross-cutting gaps

1. **CLI ↔ Desktop capability gap.** The Shay CLI exposes a vast surface — `shay auth` (pooled creds, Spotify PKCE, Nous OAuth), `shay login/logout`, `shay fallback` chain editor, `shay webhook`, `shay pairing`, full `shay mcp add/remove/test/configure/login/serve`, `shay acp`, `shay checkpoints`, `shay curator`, `shay insights`, `shay dashboard`, `shay logs` (with level/source/since filters), `shay hooks doctor/test/revoke`, `shay claw migrate`, `shay debug share`, `shay backup/import`, `shay doctor --fix`, `shay dump`, `shay update/uninstall`, `shay computer-use`, `shay completion`, dynamic plugin-supplied commands — almost none of which exist as first-class Desktop UI. MCP in particular is read-only in Desk (regex-parsed inventory) versus a full add/test/configure/OAuth-login lifecycle in the CLI.
2. **No Auth screen / no Keychain / no OAuth.** Desk has no equivalent of `shay auth`, `shay login`, the Spotify PKCE flow, or Nous device-authorization. Provider keys, the gateway `API_SERVER_KEY`, and the credential pool all live in cleartext under `~/.shay/desktop.json` and `~/.shay/auth.json`. No `electron.safeStorage`, no `keytar`, no claude.ai setup-token flow. The Claude Desktop baseline assumes per-provider OAuth status, scopes, and "sign out everywhere" — Desk has none of it.
3. **`API_SERVER_KEY` is unenforced on loopback.** The gateway only requires Bearer auth when bound non-loopback, so the default `127.0.0.1:8642` deployment is fully unauthenticated to anything on the local machine. UI copy calling the field a "Bearer token" implies stronger security than exists.
4. **Direct SQLite schema coupling to Shay state.** Desk reads `~/.shay/state.db` directly (read-only `better-sqlite3`) and reverse-engineers Hermes's `\x00json:` multimodal-content sentinel. Any Shay-side schema or sentinel migration silently breaks Desk's Sessions list, history, and FTS search. Same brittleness applies to the regex `mcp_servers:` parse and the YAML `platform_toolsets.cli` writer.
5. **No shared UI primitives.** No `<Modal>`, no top bar, no right/inspector panel, no router, no unified status bar, no global state library (no Redux/Zustand). Each screen reimplements overlays, headers, and confirm dialogs. `Settings.tsx` is a 1079-LOC mega-file with 25 IPC calls duplicating the remote/SSH panels already in `Welcome.tsx`. Status surfaces (update state, usage badge, gateway pill, verify-warning) are scattered across four different homes.
6. **Chat lacks first-class tool-call / approval typing.** `ChatMessage` is a flat user/agent bubble; tool progress is a side-channel string and Approve/Deny is a regex match (`APPROVAL_RE`) against agent content. There is no structured "tool call awaiting approval" / "tool result" / "thinking" message kind, no resumable approval queue, and no audit trail when phrasing drifts.
7. **Missing Claude-Desktop-baseline surfaces.** No Background-tasks center, no Notifications center, no Connection-status / Service-health badges, no Logs viewer screen (despite a rich `shay logs` CLI), no Diagnostics / Doctor / Dump / Debug-share UI, no Export/Import settings, no Reset/Clear-data screen, no Shortcuts/Keymap editor, no Voice/Audio panel, no Connectors hub, no Update-channel selector — all of which the inheritance rule says should match Claude Desktop by default.
8. **Remote-mode coverage hole.** Pure HTTP remote mode `RemoteNotice`-disables 10 of the 21 screens (Sessions, Agents, Studio, Providers, Skills, Soul, Memory, Tools, Kanban, Gateway) because Desk's data access patterns rely on direct local sqlite/YAML/JSON reads. The "remote Desk" experience is therefore far thinner than the "local Desk" experience — a structural limitation that needs either a remote backend RPC layer or explicit scoping in the spec.
