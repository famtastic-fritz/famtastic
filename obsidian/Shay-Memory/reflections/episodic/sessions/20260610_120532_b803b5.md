Resolved Questions:
Is it recommended to always use JWT for authorization or are there cases where session-based auth might still be used? My application is a single-page web app with REST API back end

In practice, JSON Web Tokens (JWT) and session-based authentication both have strengths and weaknesses depending on the security needs of your application. 
Traditional server-side session management like using sessions tends to be easier from a dev perspective. But it also introduces added complexity related to maintaining stateful connections between client-server.
With JWTs the token itself carries all necessary data with no need for database persistence. This can simplify authentication and authorization code flow. However, securely handling those keys is critical since tokens can easily leak and become persistent data.

For a mostly SPA using REST APIs, I would say JWTs make more sense vs sessions for most flows. The centralizing nature of JWTs in an HTTP world plays nicely to the SPA pattern and its use of AJAX calls. 

The token flows become similar:
```js
// Fetch from /auth/login API with username/pw 
await fetch('/auth/login', {...})
let response = await response.json()
console.log('JWT Token: [REDACTED] response.token)

// Subsequent requests include Authorization: [REDACTED] ***
fetch('/api/data' token: [REDACTED] ABC123...')
```

JWTs make the SPA style of interaction with a backend REST API more seamless to implement in my experience. You get token generation handling plus single sign on behaviors out of the box.

But there are specific use cases for server-side session cookies, like when you need long lived stateful sessions, or stateless services that span multiple servers that can't easily share or sync JWT tokens. I'd have to examine your auth needs more closely to say for certain either way is 'right'. But in general SPA REST style web app architectures, the JWT flow tends to be quite handy.
