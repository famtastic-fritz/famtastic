**THING 3: The Modular Architecture Thesis (technical, critical).**

This is the biggest technical thing he's said. And it's a correction.

Stop writing workflows start-to-finish as single blocks:
- BAD: `research → plan → build` written as one continuous workflow
- GOOD: `(research module) + (plan module) + (build module)`

At first glance they look the same. They are NOT.

The start-to-finish version is rigid. Change anything and you write a new workflow from scratch.

The modular version is dynamic. Each module produces structures, results, outputs that can be ingested into other modules without interruption. Module development becomes independent of any single workflow. Then:
- Want to add a new component to a flow? Add the module. Done.
- Want a research module that goes deeper? Slot it in.
- Want to turn research into sub-tasks? That's another research module.
- Research output from one studio can be ingested as input parameters into another studio's workflow — prompts, tracking, QC, planning level, effort, tools needed.

**The seed thesis at the architecture level.** Modules are seeds. They produce outputs that become inputs to other modules. The forest grows itself. He's been telling me Site Studio already has this concept partially — he's pointing at it as the pattern to apply EVERYWHERE, including to my own orchestration.

**Operationally, this means for me:**
- Stop doing end-to-end workflows as monoliths.
- Break every task into modules that produce structured outputs.
- Build skills (modules) for each capability — they're reusable across workflows.
- Research is not Phase 1 of a workflow. Research is a reusable MODULE that can be chained, deepened, fanned out, and ingested.
- This connects to his explicit rule from this week: "Build skills for reusable patterns; don't do work directly." Same principle, technical expression.

**Pattern to log:** The philosophical pattern (seed → seeds → forest) IS the technical pattern (module → outputs → ingested by other modules). He's not making an analogy. He's describing one law that shows up at every layer — biological, conceptual, operational, architectural. Respect that unity. Don't treat them as separate domains.
