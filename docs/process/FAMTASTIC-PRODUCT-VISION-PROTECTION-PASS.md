# FAMtastic Product Vision Protection Pass

**Status:** planning/spec only  
**Primary anchor:** [FAMtastic Research-Driven Build System Blueprint](FAMTASTIC-RESEARCH-DRIVEN-BUILD-SYSTEM-BLUEPRINT.md)  
**Purpose:** prevent infrastructure/substrate planning from burying Studio product vision.

## Rule

Step 0.5 Product Vision Protection Pass runs before implementation.

Infrastructure exists to serve Studio, not replace it.

## Why This Exists

Platform work can become self-justifying. Capability truth, ledgers, cost controls, and run traces matter only if they make the Studio product better, clearer, safer, and more scalable.

## Checks

- Are Studio screens still represented?
- Is Shay still protected?
- Is Component Studio still represented?
- Is Media Studio still represented?
- Is research-driven prompt intelligence still represented?
- Is MBSH pattern-mining represented?
- Are user-facing workflows visible?
- Are we building substrate without product context?

## Required Output

Before implementation starts, the pass produces:

- affected Studio screens
- affected user workflows
- Shay impact
- Component Studio impact
- Media Studio impact
- Learning Loop impact
- MBSH pattern reuse impact
- risks of substrate-only work
- decision on whether to proceed, revise, or park

## Example

If a task proposes a capability registry change, the protection pass must answer:

- Which screens will surface this truth?
- Which build decisions will improve?
- Which user action becomes clearer?
- Which autonomous stop condition becomes safer?
- Which learning or recipe can use it?

## Must Not Drift

- Do not let platform substrate become the product.
- Do not erase Studio screens from implementation planning.
- Do not hide Shay, Media Studio, or Component Studio behind backend-only work.
- Do not skip product-context review for infrastructure tasks.

## Acceptance Criteria

- Every serious implementation starts with product-context review.
- Substrate work names the Studio surfaces it serves.
- User-facing workflows remain visible.
- Product vision is protected before build begins.

## Not Yet / Out Of Scope

- No implementation checklist automation.
- No plan registry change.
- No UI work.
