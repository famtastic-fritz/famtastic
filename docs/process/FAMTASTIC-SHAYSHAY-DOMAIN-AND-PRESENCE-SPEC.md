# FAMtastic ShayShay Domain And Presence Spec

**Status:** planning/spec only  
**Primary anchor:** [FAMtastic Research-Driven Build System Blueprint](FAMTASTIC-RESEARCH-DRIVEN-BUILD-SYSTEM-BLUEPRINT.md)  
**Purpose:** preserve Shay as more than a bubble.

## Core Definition

Shay is Studio-wide assistant / guide / work partner.

Shay is distinct from public Site Assistants.

## User Decision

Shay must be visible/findable in navigation or a dedicated home/domain. Do not only protect her architecture invisibly.

This supersedes earlier drift toward "ambient only" placement. Shay can still be ambient, but she also needs an intentional home.

## Shay Surfaces

- Shay Bubble
- Shay Desk
- Shay Domain/Home
- Shay Tasks
- Shay Memory
- Shay Training
- Shay Skills
- Shay Agent Handoffs
- Shay Settings
- Shay Body / Presence modes

## Shay Bubble

Ambient contextual help anywhere.

The bubble handles:

- quick asks
- contextual explanation
- lightweight guidance
- handoff into Shay Desk when work becomes larger

## Shay Desk

Larger workbench for:

- drafts
- tasks
- handoffs
- proof
- memory candidates
- plan review
- learning candidates
- agent routing

## Shay Domain/Home

Dedicated place to manage:

- Shay tasks
- memory
- training
- body/presence
- skills
- settings
- handoffs
- queued guidance

The exact nav label remains open. The requirement is findability.

## Body / Presence Modes

- orb
- small avatar
- full body
- themed outfits
- workspace-specific roles
- public-site assistant mode is separate

Presence modes are UI/product decisions, not the same thing as assistant architecture.

## Shay vs Site Assistants

| Surface | Scope | Example |
| --- | --- | --- |
| Shay | Studio-wide, cross-project, memory/task/handoff guide | helps Fritz run builds and understand proof |
| Site Assistant | public site component, bounded by one site | Hi-Tide Harry on MBSH |

Do not let public assistant needs constrain Shay's Studio role.

## Must Not Drift

- Do not reduce Shay to a floating widget.
- Do not confuse Shay with Hi-Tide Harry or site assistants.
- Do not hide Shay's tasks, memory, skills, and handoffs in invisible backend state.
- Do not let the exact nav label block the requirement that Shay be findable.

## Acceptance Criteria

- Shay has bubble, desk, and domain/home surfaces.
- Shay is clearly separate from public Site Assistants.
- Body/presence modes are captured as future product surface options.
- Earlier ambient-only placement is explicitly corrected.

## Not Yet / Out Of Scope

- No avatar or body design.
- No Shay UI implementation.
- No memory schema change.
- No public Site Assistant work.
