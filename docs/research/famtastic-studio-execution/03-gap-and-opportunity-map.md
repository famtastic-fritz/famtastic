# FAMtastic Studio Intelligence Run - Gap and Opportunity Map

**Status:** complete  
**Purpose:** Identify where FAMtastic Studio can be meaningfully different instead of becoming another generic AI app builder.

## Gap 1: Builders Generate Before They Understand

Most builders can turn a prompt into an app quickly. Few require research, opportunity finding, positioning, and prompt strategy before generation.

**FAMtastic opportunity:** Make intelligence the first artifact. The output is not "a site"; it is a researched build recipe with proof criteria.

**V1 adaptation:** Intelligence Brief and Recipe Composer must exist before broad build automation.

## Gap 2: Preview Is Treated Like Completion

Live preview is useful, but preview success does not prove auth, DB, deploy, domains, security, accessibility, or production readiness.

**FAMtastic opportunity:** Separate preview from launch proof.

**V1 adaptation:** QA Board and Deploy Center must require proof packets and capability probes.

## Gap 3: Cost Is Often Visible After the Fact

Replit exposes checkpoint cost and usage dashboards, but most systems do not make build decisions cost-aware at the product level.

**FAMtastic opportunity:** Cost is a first-class build constraint, not billing trivia.

**V1 adaptation:** `$50+` projected actions require explicit approval; cheap/local lanes are default.

## Gap 4: Generated Assets Lose Lineage

AI-generated media is often copied into a project without prompt lineage, variants, rights/source notes, cleanup status, mobile crop proof, or allowed slot metadata.

**FAMtastic opportunity:** Media Library becomes an active asset registry.

**V1 adaptation:** Asset approval requires source, prompt object, QA state, allowed slots, fallback, and usage tracking.

## Gap 5: Component Reuse Is Weak

Builders can create components but rarely manage lifecycle: candidate -> sandbox -> proofed -> approved -> library -> installed.

**FAMtastic opportunity:** Component Studio becomes a production workflow, not a code snippet drawer.

**V1 adaptation:** Every reusable component needs metadata, allowed slots, QA gates, fallback, and usage history.

## Gap 6: Agents Have Status, But Not Product Meaning

Agent systems show task progress, logs, branches, and PRs. They do not automatically explain what a run proved, fixed, or added.

**FAMtastic opportunity:** Every pass has a closeout packet: fixes, additions, proofs, unresolved risk, learning candidates.

**V1 adaptation:** Run Control and Learning Board must answer "what did this prove?"

## Gap 7: Orchestration Frameworks Are Not Product UX

CrewAI, LangGraph, and AutoGen help build flows. They do not decide what Studio screens, approvals, or operator decisions should look like.

**FAMtastic opportunity:** Mission Control is the product experience; orchestration is hidden substrate.

**V1 adaptation:** Do not expose a generic graph dashboard. Surface "what needs Fritz right now?"

## Gap 8: Safety Is Usually Generic

Security guidance exists, but builders often lack domain-specific gates for generated sites: source maps, secrets, public asset leaks, DNS, payments, deploy target, data ownership.

**FAMtastic opportunity:** Capability Truth plus deploy readiness gates.

**V1 adaptation:** Production deploy requires secret scan, sourcemap/debug-artifact check, capability probes, and rollback note.

## Gap 9: Long-Running Work Lacks Durable Learning

Many tools can run in background, but the learned pattern may disappear into a PR or chat.

**FAMtastic opportunity:** Learning Board promotes reusable lessons into registries.

**V1 adaptation:** Completion requires learning capture and V1/V2 classification.

## Highest-Value Opening

The first implementable advantage is not a new site build. It is a Studio run model that makes every build legible:

1. What intelligence drove the plan.
2. What recipe was chosen and why.
3. What capabilities are actually working.
4. What agents did.
5. What proof exists.
6. What should become reusable.
