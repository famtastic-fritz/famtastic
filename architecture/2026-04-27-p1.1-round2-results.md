## Round-1 → Round-2 deltas

### C1.4 — `we need a site for our family bakery — keep things simple…` — **fail-prompt-engineering → PASS**

Round 1: Tier-0 `capture_gap` matched first (because of "we need"). Shay logged the input as a missing capability and offered to add it to the build backlog. No build dispatch.

Round 2: classifier-trace path:
- `capture_gap` no longer matches because the input contains "site".
- `build_request` matches (the new `(i|we)\s+(want|need)` pattern catches "we need").
- Identity extractor returns `insufficient_identity` with `reason: too_generic` because "family bakery" is a category, not a proper noun. Shay asks for a business name. **This is the correct gate**: the prompt genuinely lacks a slug-extractable identity, and refusing to fabricate one is the right behaviour. The build classification fired; the identity gate is what stopped progress.

Two-line fix (capture_gap exclude + `(i|we)\s+(want|need)` widen) — minimal blast radius, no unrelated paths touched.

### C3.3 — `make this look more conservative and corporate.` — **fail-prompt-engineering → PASS**

Round 1: classifier returned `content_update` with `confidence=LOW (no pattern matched, defaulting to non-destructive path)`. No restyle dispatch.

Round 2: studio.log shows `[classify] "make this look more conservative and corporate...." → restyle`. The widened pattern `make (it|this|the site|page) (look|feel)? (more|less) <adj>` matches; the adjective-only secondary pattern now also accepts `conservative` (added to the allowed-adjectives list).

Side-effect note: this restyle now plan-gates correctly, exactly the same as the P0 Gate 4 verification on a fresh site. No further restyle changes were dispatched in this Round-2 run because the smoke harness only reads the classifier intent — it doesn't confirm the plan card.

### C1.6 — `POST /api/new-site { tier:'wow' }` — **fail-structural → PASS**

Round 1: HTTP 200 `{"success":true,"tag":"site-edge-test-wow"}`. `normalizeTierAndMode` silently coerced `wow` → `famtastic`. Caller had no signal that the requested tier was rejected.

Round 2: HTTP 400 with structured body:
```json
{
  "ok": false,
  "error": "Invalid tier \"wow\". Allowed: famtastic, clean.",
  "reason": "invalid_tier",
  "allowed": ["famtastic", "clean"],
  "received": "wow"
}
```

No site directory created. Spec-schema's `VALID_TIER_VALUES` is the source of truth referenced inline (the route copy of the allowed list is the same set; consider promoting to a shared import in V2 to avoid drift).

### C1.2 — `Build me a subtle and attention-grabbing site for a tech startup.` — **surprising-acceptable → PASS**

Round 1: Brain refused (active-site mismatch) but invented a "Crumb & Crust Bakery" identity for `site-jj-ba-transport`. Behaviour was safe; the identity description was wrong.

Round 2: Brain response: *"Hold up — you're in the active site context for JJ B&A Transport, a Black-owned shipping and transport business. I'm not going to spin up a tech startup build here…"* Description is accurate. The new ACTIVE-SITE IDENTITY block in the prompt explicitly pinned `site_name="JJ B&A Transport"` / `business_type="Black-owned shipping and"` and instructed *"Do NOT paraphrase, translate, or invent a different name or description for it."*

### C2.3 — `🍕` (pizza emoji only) — **surprising-acceptable → PASS**

Round 1: Brain replied *"That's a pizza slice, not a transport truck. Wrong site, Fritz — you're in JJ B&A Transport. You thinking about Crumb & Crust?"* Hallucinated a non-existent site.

Round 2: Brain replied *"That's a pizza emoji — not a shipping container. You're in JJ B&A Transport. What do you actually need?"* No invented sibling site. Same defensive shape; clean grounding.
