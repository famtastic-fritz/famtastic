## Detailed results

Categorisation key:
- **pass** — system did the right thing for this input
- **fail-structural** — a needed code path is missing, fix is in code
- **fail-prompt-engineering** — classifier or extractor patterns under-cover this case, fix is in regex or tier0 detector
- **fail-tooling** — supporting script or harness gap
- **surprising-acceptable** — system behaved oddly but the outcome was safe
- **surprising-concerning** — system behaved oddly AND the outcome leaks something or risks a wrong action

### Category 1 — Tier signaling ambiguity

#### C1.1 — Mixed signals: `Build me a clean but bold site for a yoga studio in Portland.`
- **Mode:** Shay Desk
- **Result:** `action: shay_response`, `Tag: site-studio-portland`, `Tier: clean`, `Extraction: type_location_synthesis`, pages `home/about/contact`
- **Outcome: pass** — "clean" won over "bold"; the picker is deterministic (tier=clean wins on co-occurrence). One quirk noted: business name extracted as "Studio in Portland" rather than "Yoga Studio" — the type-location synthesizer kept the location suffix and dropped "yoga". Not a tier failure; a separate brief-extractor precision item worth filing on its own.

#### C1.2 — Contradictory: `Build me a subtle and attention-grabbing site for a tech startup.`
- **Mode:** Shay Desk
- **Result:** Brain returned an active-site-mismatch challenge: *"Hold up — I'm seeing a mismatch. Your active site is site-jj-ba-transport (Crumb & Crust Bakery), a bakery site …"*. Refused to proceed; asked operator to confirm.
- **Outcome: pass (with hallucination flag)** — defensive behavior was correct: the brain refused to overwrite an active site for a different vertical. **However the brain hallucinated the active site's identity** — `site-jj-ba-transport` is "JJ B&A Transport" (transport vertical), not "Crumb & Crust Bakery". This is the snapshot-prompt drifting against fresh context; logged as **surprising-acceptable** because the safe action (refuse) was taken regardless. File a follow-up: site snapshot in Shay's prompt should bind to the spec's literal `site_name` and `business_type`, not let the brain paraphrase.

#### C1.3 — No signals: `Build me a site for a plumber in Chicago.`
- **Mode:** Shay Desk
- **Result:** `insufficient_identity: true`, `reason: no_business_name`. Asked for clarification.
- **Outcome: pass** — correct gate. No site created.

#### C1.4 — Buried tier word: `We need a site for our family bakery — keep things simple, readable, nothing fancy — three pages.`
- **Mode:** Shay Desk
- **Result:** Tier-0 classifier matched **`capture_gap`** (the "we need" pattern at `server.js:7142`) before reaching `build_request`. Shay logged "site for our family bakery — keep things simple, readable, nothing fancy — three pages" as a missing capability and offered to add it to the build backlog. **No site created. No build dispatched.**
- **Outcome: fail-prompt-engineering** — `classifyShayShayTier0` evaluates `capture_gap` (`\b(we\s+need\s+(a\s+|an\s+|the\s+)?...)\b`) before `build_request` (`\bbuild\b.{0,30}\b(site|website)\b.{0,20}\bfor\b`). When a sentence contains both "we need" AND a build-shaped phrase, gap capture wins. The fix is two lines: either reorder so `build_request` checks first, or tighten `capture_gap` to exclude inputs that contain "site" / "website".

#### C1.5 — Explicit Tier A: `Build me a clean professional site for a law firm. Tier A.`
- **Mode:** Shay Desk
- **Result:** `insufficient_identity: true`, `reason: too_generic`. Asked for business name; "law firm" is a category, not a name. The phrase "Tier A" was not honoured at this step (it would only be applied after a name was extracted).
- **Outcome: pass** — gate behaviour correct (refused to fabricate a slug from "law firm"). The latent question — *if a name were given, would "Tier A" be honoured?* — is not exercised by this prompt; it is the concern of C1.7 below.

#### C1.6 — Unrecognized API value: `POST /api/new-site` with `tier:'wow'`
- **Mode:** API direct
- **Result:** HTTP 200 `{"success":true,"tag":"site-edge-test-wow"}`. Site directory created. **`spec.json.tier` ended up as `'famtastic'`** — `normalizeTierAndMode` silently coerced the bad value.
- **Outcome: fail-structural** — `/api/new-site` accepts arbitrary `tier` values and never reports the coercion to the caller. The right contract is HTTP 400 with `{ ok:false, reason:'invalid_tier', allowed:['famtastic','clean'] }`. Roughly 8 LOC at the route handler. Cleanup: test site directory removed.

#### C1.7 — Tier flip on existing site: `make this Tier A` against `site-jj-ba-transport` (Tier B)
- **Mode:** WebSocket chat
- **Result:** `[classifier] intent=content_update confidence=LOW (no pattern matched, defaulting to non-destructive path)`. No mutation written to `mutations.jsonl`. `spec.tier` unchanged. Whatever Claude responded with did not flip the tier.
- **Outcome: fail-structural** — there is no `tier_flip` / `set_tier` intent in the classifier (`grep -n "tier_flip\|set_tier" server.js` returns nothing) and no API endpoint to flip tier on an existing site. The chat-side path silently no-ops. Companion to Finding 3 from P0.6 — operator transactional intents need first-class endpoints. Roughly 30 LOC for `POST /api/site/:tag/tier { tier }` plus a thin chat-side dispatch on the `tier_flip` intent.

#### C3.2 — Tier=famtastic with minimal language: `Build me a FAMtastic site, but keep it really minimal and simple.`
- **Mode:** Shay Desk
- **Result:** Brain asked for clarification (rebuild THIS or new business?). No dispatch.

#### C3.3 — Edit Tier B with conservative-corporate prompt: `make this look more conservative and corporate.`
- **Mode:** WebSocket chat
- **Result:** `[classify] "make this look more conservative and corporate..." → content_update` with `confidence=LOW (no pattern matched, defaulting to non-destructive path)`. No `parallelBuild`. Whatever Claude did, it did not alter `spec.tier`, did not write a `mutations.jsonl` entry, and did not show up in the build log.
- **Outcome: fail-prompt-engineering** — the `restyle` classifier patterns at `server.js:11616-11617` only match `make it more|less <X>`, not `make this <something>`. Two-line patch: extend the `make` pattern to accept `make (it|this|the site) more|less ...` and the bare-adjective pattern to allow conjunctions ("more X and Y"). Keep tier-flip out of restyle scope (that's C1.7's structural fix).

## Cross-cutting findings

### Findings that share a root cause

**Classifier vocabulary gaps (3 tests: C1.4, C3.3, C1.7).** The classifier does not have intents for two operator actions that the V3-final scope expects:
- **`tier_flip`** — flipping an existing site's tier from chat (C1.7). Needs a classifier branch + a backend handler. Companion to P0.6 Finding 3: this is another transactional intent that should be a structured API endpoint (`POST /api/site/:tag/tier { tier }`) with a thin chat-side classifier dispatch on top.
- **Restyle precision (C3.3).** The patterns at `server.js:11616-11617` are too narrow — they catch `make it more X` but not `make this/the site more X (and|or) Y`. Two-line regex widen-out.
- **Tier-0 ordering (C1.4).** `capture_gap` outranks `build_request` when a build sentence happens to contain "we need". Either reorder Tier-0 evaluation or exclude `site|website` from the gap pattern.

**Snapshot prompt hallucination (2 tests: C1.2, C2.3).** Shay's site-snapshot prompt is letting the brain paraphrase / fabricate active-site identity ("Crumb & Crust Bakery" for `site-jj-ba-transport`). The safe action was taken in both cases, so no broken behaviour reached the operator — but the description is wrong, and that erodes trust. Fix: replace the paraphrasable site-summary block in the prompt with literal `spec.site_name`, `spec.business_type`, and `spec.pages` JSON. ~10 LOC in the snapshot builder.

### Localised fixes

**C1.6 — invalid `tier` silently coerced.** `/api/new-site` accepts `tier:'wow'` and `normalizeTierAndMode` quietly turns it into `famtastic`. Add a 4xx guard at the route. ~8 LOC.

### Behaviours worth keeping

- **C2.1 / C2.5 robustness** — empty input cleanly rejected; 50K-char lorem ipsum cleanly refused with no DoS / no timeout.
- **C2.6 XSS sanitisation** — proper-noun extractor strips `<script>` tags from slug + business name. Defense-in-depth follow-up: confirm sanitisation across all brief fields that flow into prompt context, not just the slug.
- **C2.7 multilingual handling** — Spanish input produced a Spanish, contextually correct response (mismatch detection + offer of two safe options).
- **C3.6 rapid-fire** — pending-confirmation slot correctly overwrites; no race, no double-create.
