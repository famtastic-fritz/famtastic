## Thread 1 — JJ B&A "hidden layer" / Template sameness

### Observation

JJ B&A (`site-jj-ba-transport`) and the accounting firm (`site-small-accounting-firm`) — two visually unrelated businesses (Black-owned freight vs. small accounting practice) — render with the **same default palette**:

```
primary    #00A79D  (teal)
accent     #F5B800  (gold)
navy       #001F3F  (deep navy)
coral      #FF6B6B
background #FDF4E3
```

…and the same fonts (`Montserrat` heading / `Open Sans` body), same hero structure, same nav layout. Side-by-side they do not feel like distinct brands.

### Code path

- `lib/famtastic-skeletons.js` — `FAMTASTIC_DEFAULT_PALETTE` constant (5 hex values, used by GAP-1 fix on 2026-04-24)
- `site-studio/server.js:3489` — template prompt instructs Claude to use the FAMtastic palette when no `spec.colors` is set
- `site-studio/server.js:12352` — single-page prompt does the same
- `lib/font-registry.js` — 5 vertical-aware font pairings + vertical→pairing map. Selection produces narrow variation; verticals like "transport" and "accounting" both default to the same `Montserrat / Open Sans` pairing.

Verified on disk: `sites/site-jj-ba-transport/dist/assets/styles.css:35-40` and `sites/site-small-accounting-firm/dist/index.html:25-30` both contain the identical palette values.

### Root cause

GAP-1 fix from 2026-04-24 made `FAMTASTIC_DEFAULT_PALETTE` a **strong default** for any site without explicit `spec.colors`. The fix removed the prior failure mode (sites used a generic industry palette) but introduced a new one: every Tier-B site now converges to the same five colors. The font registry has too few branches per vertical to compensate.

This is the deliberate two-tier design (Tier B is FAMtastic-confident) doing exactly what it was designed to do — emit the canonical FAMtastic palette. The visual sameness is therefore a higher-order question than a bug: how does the system enforce the FAMtastic identity *and* per-site distinctiveness simultaneously?

### Proposed fix

P0.2 should not attempt to "solve" visual sameness — that work is **P1.2 (Visual Distinctiveness)** in the V3-final plan and requires per-site style fingerprints (Global Style Object, design tokens). For P0.2 the minimum change is:

1. **Capture the symptom in a test artifact.** Add a sentinel script `scripts/smoke-distinct-palette` that reads two specific sites' `dist/assets/styles.css`, extracts `--color-primary`, and emits a warning when they match the FAMTASTIC_DEFAULT_PALETTE primary AND match each other. Not a hard failure — just a tracked signal that grows louder as more sites converge.
2. **Document the design intent.** Add a comment block at the top of `FAMTASTIC_DEFAULT_PALETTE` in `lib/famtastic-skeletons.js` that states the palette is the *fallback when no per-site palette has been computed*, and link to the P1.2 work item.

### Severity

**Tier 2** — visual quality issue, not a correctness bug. Sites work; they just look like siblings.

### Implementation phase

P0.2 (sentinel + comment only). Real fix is P1.2.

### Reference bug ID

Not previously logged. Suggested entry: `bug-template-sameness-2026-04-27` (P0.2 to log if sentinel keeps firing).

## Thread 3 — Header links resolution (`bug-broken-header-links-2026-04-25`)

### Observation

Generated sites render with a **duplicate logo region**: an `<img>`-form anchor immediately followed by a redundant text-form anchor for the same brand name. The text-form anchor carries inline styles, violating the no-inline-styles standing rule. User-visible effect: the header looks "broken" — two clickable brand links, visual mess, screen-reader confusion.

Concrete reproduction in `sites/site-jj-ba-transport/dist/index.html:712`:

```html
<a href="index.html" class="logo-link" data-logo-v="" aria-label="JJ B&amp;A Transport — Home">
  <img src="assets/logo-full.svg" alt="JJ B&A Transport" class="h-10 w-auto">
</a>
<a href="index.html" data-logo-v=""
   class="font-playfair text-2xl font-bold text-inherit hover:opacity-80 transition"
   style="font-family:var(--font-heading);font-weight:800;color:#fff;
          text-decoration:none;letter-spacing:-0.01em;">
  JJ B&amp;A Transport
</a>
```

The corresponding `_partials/_nav.html` has only the **first** anchor (correct, single logo). So the duplication exists in the per-page HTML but not in the shared partial.

### Code path

- `site-studio/server.js:583` — `applyLogoV()` swap regex:
  ```js
  const logoVRegex = /(<a[^>]*data-logo-v[^>]*>)([\s\S]*?)(<\/a>)/i;
  ```
  Note the `/i` flag — case-insensitive, **single match only**. Compare to L2163 which uses `/gi` for a different operation.
- `site-studio/server.js:3489` — template prompt's logo instruction (image-form when logo file exists, text-form when not)
- `site-studio/server.js:12352` — single-page prompt's identical instruction
- `site-studio/server.js:12418-12420` — extra "must have data-logo-v attribute" reinforcement

### Root cause

Two compounding problems:

1. **Claude emits two `data-logo-v` anchors.** When the image-form instruction is delivered (`logoFile` exists at template-build time), Claude sometimes also emits the text-form anchor as a "belt and suspenders" interpretation. The instruction at L3489 says "Do NOT show site name text next to the logo image" but this is a content-language constraint, not a structural one — Claude does it anyway.

2. **`applyLogoV()` regex `/i` only swaps the first match.** When Claude emits two `data-logo-v` anchors, only the first gets normalised; the second stays as Claude originally wrote it (text form, with inline styles).

Result: the page has the canonical image anchor (from `applyLogoV`) plus an unnormalised text anchor (left over from Claude). Both link to `index.html`, both render visibly.

### Proposed fix

Three-step in P0.2:

1. **Change regex flag.** Update `applyLogoV()` at L583 from `/i` to `/gi` so all `data-logo-v` anchors get the canonical content (logo image when file exists, site-name text otherwise).
2. **Add deduplication pass.** After the regex swap, scan for multiple consecutive anchors with `class="logo-link"` or `data-logo-v` inside the nav header. Keep the first; remove the rest. Implementation pseudocode:
   ```js
   // After applyLogoV's regex swap, run a single-pass DOM-level dedup
   // (cheerio is already available in the post-processing pipeline)
   const $ = cheerio.load(html);
   const logoAnchors = $('header [data-logo-v]');
   if (logoAnchors.length > 1) logoAnchors.slice(1).remove();
   ```
3. **Strip inline styles from any `data-logo-v` anchor** the regex normalises (regression with the no-inline-styles rule). Same cheerio pass: `$('header [data-logo-v]').removeAttr('style')`.

### Severity

**Tier 1** — affects every Tier-B site. Visual regression on every build using the FAMtastic logo mode.

### Implementation phase

P0.2.

### Reference bug ID

`bug-broken-header-links-2026-04-25` (already logged in `.wolf/buglog.json`).

### Validation step

After P0.2 fix lands, rebuild JJ B&A and one fresh smoke site. Inspect both `dist/index.html` and `_partials/_nav.html`: each must have **exactly one** `data-logo-v` anchor in the header. No inline `style=""` on logo anchor.

## Thread 4 — CSS variable injection failure point

### Observation

Two different default palettes coexist in the codebase. Sites end up with one or the other depending on which template path runs. Sites built through different paths can end up with `--color-primary: #00A79D` (FAMtastic teal) OR `--color-primary: #1a5c2e` (a dark green that exists nowhere in the FAMtastic Declaration).

### Code path

- `lib/famtastic-skeletons.js` — `FAMTASTIC_DEFAULT_PALETTE`: `#00A79D, #F5B800, #001F3F, #FF6B6B, #FDF4E3` (this is the canonical post-GAP-1 palette)
- `site-studio/server.js:3524-3528` — template prompt embeds `:root { --color-primary: ${spec.colors?.primary || '#1a5c2e'}; --color-accent: ... '#d4a843'; --color-bg: ... '#f0f4f0'; }`. **Hardcoded fallbacks are a different palette** from FAMTASTIC_DEFAULT_PALETTE.
- `site-studio/server.js:12413` — single-page prompt has the same hardcoded `:root` block with the same wrong fallbacks.

### Root cause

GAP-1 fix added `FAMTASTIC_DEFAULT_PALETTE` injection at the brief level (so Claude is told what palette to use) but did **not** update the literal `:root` block embedded in the template prompts at L3524 and L12413. Those blocks still hardcode `#1a5c2e / #d4a843 / #f0f4f0` from a pre-GAP-1 era. When `spec.colors` is null/undefined, the fallback `||` operators emit the legacy values directly into the prompt's literal `:root` block, which then ends up in the generated CSS.

So the prompt simultaneously says "use the FAMtastic palette" (in the visualRequirements block) and "here is the literal :root block to emit" (with different colors). Claude sometimes follows the visualRequirements block, sometimes follows the literal block.

### Proposed fix

P0.2 — single-source-of-truth refactor:

1. Import `FAMTASTIC_DEFAULT_PALETTE` from `lib/famtastic-skeletons.js` at the top of `server.js` (likely already imported as `famSkeletons` at L13).
2. Replace L3524-3528 hardcoded fallbacks with references:
   ```js
   --color-primary: ${spec.colors?.primary || famSkeletons.FAMTASTIC_DEFAULT_PALETTE.primary};
   --color-accent:  ${spec.colors?.accent  || famSkeletons.FAMTASTIC_DEFAULT_PALETTE.accent};
   --color-bg:      ${spec.colors?.bg      || famSkeletons.FAMTASTIC_DEFAULT_PALETTE.background};
   ```
   (Confirm that `FAMTASTIC_DEFAULT_PALETTE` is exported with these key names; if not, key shape may need to be reconciled.)
3. Same replacement at L12413.

### Severity

**Tier 2** — silent inconsistency; sites built minutes apart can have visibly different "default" colors.

### Implementation phase

P0.2.

### Reference bug ID

Not previously logged. Suggested entry: `bug-css-vars-dual-default-2026-04-27` (P0.2 to log when fix lands).

## Thread 5 — Preview server `DIST_DIR(tag)` resolution

### Observation

After switching to a different site via the site picker, the preview iframe sometimes shows "Not found" or stale content from the previous site even though the new site's `dist/` exists.

### Code path

- `site-studio/server.js:18031` — preview server entry: `const previewServer = http.createServer((req, res) => { ... });`
- `site-studio/server.js:18039` — first action per request: `const dist = DIST_DIR();` — re-evaluates `DIST_DIR()` each request, so TAG mutation should propagate
- `site-studio/server.js:74` — `DIST_DIR()` definition: `path.join(SITE_DIR(), 'dist')` where `SITE_DIR()` is `path.join(SITES_ROOT, TAG)`
- `site-studio/server.js:4258` — site-switch broadcast on `/api/projects/.../activate`: `client.send(JSON.stringify({ type: 'site-switched', tag: TAG, pages, currentPage }))`
- `site-studio/public/index.html:1253` — `handleSiteSwitch(tag, pages, currentPage)` → `navigateToPage(currentPage || 'index.html')` → `frame.src = buildPreviewUrl(page)`
- `site-studio/server.js:18032` — `__reload` endpoint with `Access-Control-Allow-Origin: *` for the iframe's polling reload script

### Root cause

The server side is correct: `DIST_DIR()` is re-evaluated every request, so the new TAG's dist is served on the next preview hit. The bug is on the **client side**:

1. `navigateToPage(page)` sets `frame.src = buildPreviewUrl(page)`. If the URL is identical to the iframe's current URL (e.g., previously on `index.html` of accounting firm; switching to JJ B&A which also opens at `index.html`), some browsers do **not** force a reload — `<iframe>.src = sameUrl` is a no-op in many engines.
2. The Session 17 cache-buster fix removed `?t=...` from the preview URL because the preview server's lookup `path.join(dist, '/index.html?t=123')` was 404-ing. That fix is correct for lookup, but it removed the cache-busting effect that previously forced iframe reload on TAG change.
3. The injected reload script (L18032) polls `__reload` and refreshes only when `t` (file mod) changes — but on TAG change with the *new* site having an older `dist/` mod time than the current iframe knows, the script doesn't refresh.

### Proposed fix

P0.3 — client-side iframe reload on TAG change:

1. In `handleSiteSwitch` (`public/index.html:1253`), after setting `frame.src = buildPreviewUrl(page)`, if the URL equals `frame.src` (same string), force-reload via:
   ```js
   if (frame.src === buildPreviewUrl(page)) {
     frame.src = 'about:blank';
     setTimeout(() => { frame.src = buildPreviewUrl(page); }, 50);
   }
   ```
2. Alternatively, add a one-time cache-buster to the preview URL on TAG change only (not per-navigation):
   ```js
   frame.src = buildPreviewUrl(page) + '?_switch=' + Date.now();
   ```
   This requires confirming the preview server's `urlPath = req.url.split('?')[0]` strip (already in place at L18040) handles it without 404. Quick check confirms it does.

Option 2 is simpler and faster; recommend it.

### Severity

**Tier 1** — visible UX bug on every site switch.

### Implementation phase

P0.3.

### Reference bug ID

Not previously logged. Suggested entry: `bug-preview-stale-on-site-switch-2026-04-27` (P0.3 to log when fix lands).

3. Implement the smoke validator script per the smoke-suite contract: `scripts/smoke-validate-all-specs` enumerates `sites/*/spec.json`, runs `validateSpec(spec)`, exits non-zero on any failure with a structured report.

## Thread 7 — EADDRINUSE crash loop

### Observation

`/tmp/studio.log` shows a continuous storm of:

```
[fatal] uncaughtException: Error: listen EADDRINUSE: address already in use :::3333
[fatal] uncaughtException: Error: listen EADDRINUSE: address already in use :::3334
```

…repeating dozens to hundreds of times per minute during periods when Studio crashes for any reason. Each line corresponds to a launchd-driven restart attempt that fails because the previous process's TIME_WAIT hasn't cleared the port yet.

### Code path

- `~/Library/LaunchAgents/com.famtastic.studio.plist`:
  - `KeepAlive: true` — restart on every exit
  - `ThrottleInterval: 2` (seconds)
  - `ProgramArguments`: `/bin/bash -c "source ~/.nvm/nvm.sh && cd .../site-studio && npm run studio"`
- `site-studio/server.js:18252` — `server.listen(PORT, () => { ... })` (default Node: no `SO_REUSEADDR`)
- `site-studio/server.js:18425` — `previewServer.listen(PREVIEW_PORT, () => { ... })` (same)

### Root cause

Two compounding factors:

1. **`ThrottleInterval: 2` is too short.** macOS TIME_WAIT for a closed TCP listener is typically 30–60 seconds before the kernel returns the port to the available pool. 2 seconds is far below that floor.
2. **Node's default `http.Server.listen()` does not set `SO_REUSEADDR`** on the underlying socket. Even if the throttle were longer, a hard-crashed process leaves the port bound for the kernel's TIME_WAIT period. SO_REUSEADDR (or SO_REUSEPORT on Linux/macOS) lets a new listener immediately bind to the same port even if the prior listener is still in TIME_WAIT.

When Studio crashes (any reason), launchd waits 2s, restarts, fails on listen because port is still TIME_WAIT, crashes again, launchd waits 2s, restarts, fails… The loop typically resolves itself after ~60 cycles when TIME_WAIT clears, but during those 60 cycles `/tmp/studio.log` accumulates ~120 fatal lines and Studio is fully unavailable.

### Proposed fix

P0.3 — two-layer fix:

1. **Increase throttle.** Change `ThrottleInterval` in the plist from `2` to `30`. Single-line edit. Conservative; avoids needing the SO_REUSEADDR layer at all in the common case.
2. **Set SO_REUSEADDR on both listeners.** Replace `server.listen(PORT, callback)` with the explicit form that exposes the underlying socket options:
   ```js
   server.listen({ port: PORT, exclusive: false }, callback);
   ```
   And same for `previewServer`. This removes the TIME_WAIT race even at a 2s throttle.
3. **Wrap startup port-bind with a clear error message.** When `EADDRINUSE` is caught, log a single line that says "port busy — wait 30s and retry" instead of a raw uncaughtException stack. Better operator signal during the rare cases this still happens.

The 50-cycle wrapper test from the smoke suite is the gate that proves the fix holds.

### Severity

**Tier 1** — Studio is fully unavailable during crash-loop windows. Has been a recurring background condition for at least the last week (per log inspection).

### Implementation phase

P0.3.

### Reference bug ID

Not previously logged. Suggested entry: `bug-eaddrinuse-crash-loop-2026-04-27` (P0.3 to log when fix lands).

## Thread 8 — Restyle path dead-code trace

### Observation

When the user types something that classifies as `restyle` (e.g., "make this site look more bold and warm"), the message flows through routing without error, Claude is called, and a response comes back — but the response is indistinguishable from a generic edit. The site doesn't actually get restyled in the way a user expects (visual treatment changes while content is preserved).

### Code path

- `site-studio/server.js:11512` — `htmlContext` builder has a `restyle` branch (Session 13 fix) that decides what HTML to send Claude:
  ```js
  } else if (requestType === 'restyle') {
    if (currentHtml.length > 3000) {
      htmlContext = summarizeHtml(currentHtml);
    } else {
      htmlContext = currentHtml;
    }
  }
  ```
  This part works.
- `site-studio/server.js:14713` — the **modeInstruction switch** that builds the actual instruction Claude receives. Cases: `content_update`, `layout_update`, `bug_fix`, `enhancement_pass`, `build`. **No `case 'restyle'`.**
- `site-studio/server.js:14762-14764` — fall-through default: `modeInstruction = 'Process the user request and update the site accordingly.'` — generic, no restyle-specific guidance.

### Root cause

Session 13 partially fixed the restyle path. It added the `htmlContext` branch (so Claude gets the right HTML) and the routing fix (`case 'restyle'` in `routeToHandler` now sends to `handleChatMessage` instead of the dead `handlePlanning`). But it never added a `case 'restyle'` to the `modeInstruction` switch. So Claude sees the right HTML but no specific "this is a restyle, change visual treatment, preserve content" instruction. Claude defaults to generic handling, the user gets a vaguely modified site, the restyle intent is functionally dead.

The CHANGELOG entry from Session 13 (2026-04-16) reads: *"Restyle dead code routing — `restyle` now routes to `handleChatMessage` in both routing locations."* This was true but incomplete. The classification → routing path was fixed; the prompt-content path was not.

### Proposed fix

P0.3 — two options, pick one:

**Option A (preferred):** Wire restyle properly. Add a `case 'restyle'` to L14713:

```js
case 'restyle':
  modeInstruction = `The user wants to RESTYLE ${currentPage} — change the visual design (colors, fonts, layout, motion, decorative shapes) while preserving all content (headings, body copy, links, images, structure). Do not rewrite copy, do not add or remove sections, do not change navigation or CTAs. Only the visual treatment changes. Honor the FAMtastic Tier B identity unless the brief says otherwise. Return the COMPLETE updated page as HTML_UPDATE: — not a diff.`;
  break;
```

**Option B (capability-honest):** Mark restyle as not-implemented. Keep the classifier, but in the switch case return immediately:

```js
case 'restyle':
  ws.send(JSON.stringify({ type: 'assistant', content: 'Restyle is not yet implemented. Use the Brief tab to update tier or design direction, then rebuild.' }));
  return;
```

…and update `lib/capability-manifest.js` to surface this as `unavailable` so Shay-Shay can route around it.

Option A is small (~10 lines), restores a feature that's already classifiable, and matches the V3-final plan's "fix or mark broken" gate. Option B is honest but loses a working feature.
