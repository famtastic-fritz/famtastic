# P0.6 Diagnostic — Three Silent-Dispatch Failures

**Date:** 2026-04-27
**Status:** Diagnosis only. **No code changes made.**
**References:**
- `architecture/2026-04-27-p0-exit-gate-results.md`

## TL;DR — common architectural root cause

All three findings are instances of **the same architectural meta-bug**:

> **Every operator transactional affordance (deploy button, rebuild button, content edit) is funneled through a *conversational* path (chat WebSocket / Shay Desk brain), and there is no contract that guarantees the dispatcher actually fired a handler.** The chat path silently no-ops in three independent ways — (a) by routing to a brain that hallucinates the action, (b) by hitting the brain-route gate at `server.js:16040-16048` that drops non-build intents into `handleBrainstorm`, (c) by the deploy preflight returning early with no log line. None of these surface "the dispatch failed" to the operator.

The highest-priority architectural fix is therefore **not** to patch each finding individually but to **promote operator-transactional actions to first-class HTTP endpoints (`POST /api/deploy`, `POST /api/rebuild`, `POST /api/content-edit`)** and have UI buttons + Shay Desk both target those endpoints. A 200 response means the dispatcher actually fired; a 4xx/5xx means it didn't, with a structured reason. The chat path remains the right surface for free-form natural language, but it stops being the load-bearing rail for buttons.

I'd treat that promotion as a **P0.6 hotfix-shaped change** for at minimum `POST /api/deploy` (smallest scope, biggest unblock) and defer `/api/rebuild` and `/api/content-edit` to **V2** when the broader chat-vs-API split is reconciled.

Per-finding details below.

## Finding 1 — Shay Desk doesn't dispatch content edits

### Symptom

User pastes `change the hero headline to 'Bloom Where You're Planted'` into Shay Desk. Shay's chat bubble: `On it — reading the current hero section now.` Then nothing. No edit applied. Same prompt in Studio Chat works.

### Root cause

**Shay Desk has no `content_update` (or any edit) intent in its dispatch chain. The "On it…" response is a brain hallucination.**

Trace through `server.js`:

1. `app.post('/api/shay-shay'…)` at `server.js:5950`+ receives the message.
2. Tier 0 deterministic classifier: `classifyShayShayTier0(lower, context)` at `server.js:7117-7164`. The intents it can return are:
   - `confirm_build_request`, `cancel_build_request`, `studio_command`, `route_to_chat`, `show_me`, `system_status`, `suggest_brainstorm`, `capture_gap`, `create_job_plan`, `character_pipeline`, `autonomous_build`, `build_request`.
   - **There is no `content_update`, `layout_update`, `restyle`, `bug_fix`, or `deploy` branch.** None of those intents exist in Tier 0 at all.
   - For a `change the hero headline…` message, Tier 0 returns `null`.
3. Tier 0 returning null: dispatcher continues at `server.js:6031` — `answerShayShayDirect(lower, siteSnapshot, manifest)`. This is a small Q&A lookup (capabilities, mission control); it does not handle edits and returns null for this message.
4. Falls through to Tier 1-3 brain call: `executeShayShayBrainCall(...)` at `server.js:6056`. The brain (Sonnet/Haiku/Gemini/etc., depending on `selectShayShayBrain` at `server.js:6049`) generates a **conversational response**. The brain has access to a "bridge request" affordance (`server.js:6104`) that supports four ops — `read`, `write`, `exec`, `diff` (`executeBridgeOp` at `server.js:7009-7100`).
5. **Critical gap:** `executeBridgeOp` does **not** expose a "dispatch a Studio chat edit" op. The brain can read the HTML file, can in principle write it (bypassing `surgicalEditor`, `mutations.jsonl`, the preview reload chain, post-processing, structural-hint logic — none of which is desirable), or can `exec bash`. It cannot fire `handleChatMessage(ws, msg, 'content_update', spec)` from inside Shay Desk.
6. The brain's actual response is just text. "On it — reading the current hero section now." is the LLM ad-libbing what *should* happen — and may even fire `op: read` to look at the HTML — but no edit dispatch happens because that pathway literally does not exist for Shay Desk.

### Compare with `build_request` (the path that DOES forward)

`build_request` works because it is intercepted in the `/api/shay-shay` dispatcher at `server.js:6005-6011` **before** reaching the brain:

```js
// server.js:6005
if (tier0.intent === 'build_request') {
  suggestionLogger.logSuggestion(...);
  const buildResult = await handleShayBuildRequest(message, context);
  return res.json({ ...buildResult, bridge_debug: bridgeDebug });
}
```

`handleShayBuildRequest` (and on confirmation, `handleShayBuildConfirmation`) does the real work: extract brief → `createSite` → `synthesizeDesignBriefForBuild` → `triggerSiteBuild` → `routeToHandler(buildWs, 'build', ...)`. That is the contract: **deterministic Tier 0 intent + dedicated handler, no brain involvement.**

**There is no equivalent `content_edit_request` Tier 0 intent and no `handleShayContentEdit`.** That asymmetry is the root cause.

### Proposed fix location

| Element | Where | LOC |
|---------|-------|-----|
| Add Tier 0 detection | `server.js:7117-7164` `classifyShayShayTier0` — add a `content_edit_request` branch matching the same patterns the chat-side classifier uses (`server.js:11536-11538` and the `change/update/replace/edit/set/fix/correct/modify` keyword list) | ~15 LOC regex |
| Add dispatcher interception | `server.js:6005`-style block — `if (tier0.intent === 'content_edit_request') { … await handleShayContentEdit(message, context); … return; }` | ~10 LOC |
| Add handler `handleShayContentEdit` | New function near `handleShayBuildRequest` (`server.js:8272`-ish). It should construct a build-WS-like proxy (similar to `triggerSiteBuild`'s `buildWs` mock at `server.js:7665-7683`) and call `handleChatMessage(buildWs, message, 'content_update', readSpec())`. Expose progress via `ws.send` broadcast to all connected clients so the user sees the same surgical-edit progress they'd see from chat. | ~40 LOC |

**Total ~65 LOC.** The pattern already exists for build (two prior fixes wired it up); content_edit is the same shape with a different terminal handler.

### Hotfix vs deferral

**Defer to V2.** Workaround exists: users type the edit in Studio Chat instead of Shay Desk. Shay Desk works for everything else (build requests, queries, system status). Wiring `content_edit_request` cleanly requires also thinking about `layout_update`, `restyle`, `bug_fix`, `restructure` — they have the same shape. Doing one without the others is partial; doing all five is V2 scope.

## Finding 2 — `runDeploy` never invoked despite classifier `'deploy'`

### Symptom

Server log:
```
[classify] "deploy to staging..." → deploy
```
But no `[deploy]` stderr lines, no Netlify CLI invocation, no chat error message. User clicked Deploy multiple times.

### Two candidate root causes — both real, only one fires for any given click

**Cause 2A — silent deploy preflight failure (most likely the user's symptom)**

`runDeploy` at `server.js:17169-17190`:

```js
async function runDeploy(ws, env) {
  if (deployInProgress) { … return; }
  …
  let netlify;
  try {
    netlify = await checkNetlify();   // server.js:17181
  } catch (probeErr) { … }
  if (!netlify || !netlify.ok) {
    const detail = (netlify && netlify.details) || 'Netlify is not configured.';
    try { ws.send(JSON.stringify({ type: 'error', content: `${envLabel} deploy failed: ${detail}` })); } catch {}
    appendConvo({ role: 'assistant', content: `${envLabel} deploy preflight failed: ${detail}`, … });
    return;
  }
  …
}
```

`checkNetlify` (`lib/capability-manifest.js:57-101`) returns `{ ok:false, reason:'cli_missing', details:'Netlify CLI not found on PATH…' }` on this machine. **`runDeploy` then `return`s without ever logging to stdout/stderr.** The only observable signal is the WS `error` message and a `conversation.jsonl` append.

So `runDeploy` IS firing — it just produces no log line because:
- there's no `console.log('[deploy]')` at the top of the function (only `console.error('[deploy]', trimmed)` on line 17226 inside the child stderr handler)
- the early-return path on preflight failure is silent on stdout/stderr

If the user is reading `/tmp/studio.log`, they see nothing past the `[classify]` line. The error went only to the WS and to `conversation.jsonl`. The exit-gate report explicitly required *structured reasons* on log/UI for Path B; that surface is half-implemented (WS message exists; log line does not).

The fact that the user clicked repeatedly without a chat error suggests the WS error message either isn't rendering in the active chat tab, or got rendered as a `type: 'error'` ws message which the chat panel may treat differently from `type: 'assistant'`. Worth confirming with browser devtools — but irrespective, the logging gap is real.

**Cause 2B — non-Claude brain swallows `'deploy'` into `handleBrainstorm`**

`server.js:16040-16048`:

```js
if (currentBrain !== 'claude') {
  const nonBuildIntents = ['brainstorm', 'content_update', 'bug_fix'];
  const quickClassify = classifyRequest(userMessage, spec);
  const isBuildIntent = ['build', 'layout_update', 'restyle', 'restructure',
                         'major_revision', 'new_site', 'enhancement_pass'].includes(quickClassify);
  if (!isBuildIntent) {
    console.log(`[brain-route] ${currentBrain} selected — routing "..." to handleBrainstorm`);
    handleBrainstorm(ws, userMessage, spec);
    return;
  }
  …
}
```

`'deploy'` is not in the `isBuildIntent` whitelist. So when Studio's status bar shows any non-Claude brain selected (Sonnet, Gemini, Codex, …) — which is the normal case at the moment — `'deploy'` is silently routed to `handleBrainstorm` and the official `[classify]` log line at `server.js:16057` never prints.

The user's symptom (`[classify] → deploy` did print, then silence) means **2B did not fire on that specific click** (so currentBrain was 'claude'), but 2B is a **second silent-dispatch failure waiting to fire** the moment the user picks a non-Claude brain. The same gate also drops `'page_switch'`, `'git_push'`, `'hub_push'`, `'query'`, and `'asset_import'` into the brainstorm handler.

**The whitelist at line 16043 is wrong-shape:** it lists *intents that need Claude for HTML generation* and routes everything else to brainstorm — but `'deploy'`, `'git_push'`, `'hub_push'`, `'page_switch'` are intents that need *neither Claude nor brainstorm* — they have direct handlers in the dispatcher (`server.js:16605, 16614, 16620, 16084` respectively). They should pass through unconditionally.

### Proposed fix locations

| Cause | Where | Fix | LOC |
|-------|-------|-----|-----|
| 2A — runDeploy silent preflight | `server.js:17169` add `console.log('[deploy] starting env=' + env)` at the top, and `console.log('[deploy] preflight failed reason=' + netlify.reason + ' details=' + detail)` before the early return at `:17189` | observability, no behavior change | ~3 LOC |
| 2A — chat-side surfacing | confirm `type:'error'` WS messages render in the chat panel (UI side, in `public/js/studio-shell.js` or similar) — separate scope | UI verify | n/a (verify-only) |
| 2B — brain-route whitelist | `server.js:16043` widen the bypass list to also pass through direct-dispatch intents: `'deploy', 'git_push', 'hub_push', 'page_switch', 'query', 'visual_inspect', 'asset_import'`. Or invert the gate: define a `BRAINSTORM_INTENTS = ['brainstorm', 'content_update', 'bug_fix']` set and only divert those, letting everything else fall through. The inverted-gate version is more honest about what's actually being routed away from. | array edit | ~5 LOC |

**Total ~8 LOC for both observability + whitelist fix.**

### Cross-reference to V8/V9 baseline closure plan

V8/V9 Workstream 3 (the deploy-failure-surfacing work, commit `35d73b9 feat(deploy): structured failure reasons and chat session breaks`) wired the *structured reason* into `runDeploy`'s WS message and `conversation.jsonl` entry. It did **not** add corresponding stdout/stderr `console.log` lines, and did not inventory whether all reasons render in the active chat panel. The V8/V9 work closed Path-B's *data* surface but not the *log* surface — exactly the gap surfaced today. The brain-route gate at `server.js:16040` predates V8/V9 and was never reconsidered against the deploy-via-chat assumption. This is a partial closure.

### Hotfix vs deferral

**Cause 2A logging additions: ship in P0.6 hotfix.** ~3 LOC, observability-only, zero risk.

**Cause 2B brain-route whitelist: ship in P0.6 hotfix.** ~5 LOC. The current whitelist is genuinely incorrect for any direct-dispatch intent, not just `'deploy'`. Without this fix, switching to a non-Claude brain silently breaks Deploy, page navigation, and git push.

**Confirming the WS error-message rendering: ship in P0.6 verification, not code.** Operator opens DevTools, clicks Deploy, confirms the `type:'error'` message appears in the chat panel. If yes, no UI change. If no, that's a separate UI patch.

## Finding 3 — Deploy buttons fire chat messages

### Symptom

The Deploy → Staging button posts the literal text `deploy to staging` as a chat WebSocket message instead of calling a deploy API directly. (Same pattern for → Production and Rebuild.)

### Trace

`site-studio/public/index.html:1722-1740`:

```html
1724  function deployToStaging() {
1725    if (ws && ws.readyState === WebSocket.OPEN) {
1726      addMessage('user', 'Deploy to staging');
1727      ws.send(JSON.stringify({ type: 'chat', content: 'deploy to staging' }));
1728      steps = []; stepStart = null;
1729      addStep('Deploying to staging...');
1730    }
1731  }

1733  function deployToProduction() {
1734    if (ws && ws.readyState === WebSocket.OPEN) {
1735      addMessage('user', 'Deploy to production');
1736      ws.send(JSON.stringify({ type: 'chat', content: 'deploy' }));
1737      …
1738    }
1739  }
```

Companion `rebuildSite()` at `index.html:1345`: `ws.send({type:'chat', content:'Rebuild the site'})`.

These three buttons all dispatch by *speaking the action into chat as if the user typed it*. That makes them susceptible to:

- the classifier returning anything other than the expected intent (P0 exit gate, before P0.4)
- the brain-route gate at `server.js:16040` (Finding 2, Cause 2B)
- `runDeploy`'s silent preflight return (Finding 2, Cause 2A)
- any future change to the classifier that adjusts intent precedence

### What proper API endpoints should be called

There is no `POST /api/deploy` endpoint. Closest neighbours: `GET /api/deploy-info` at `server.js:1085` (read-only), `POST /api/autonomous-build` at `server.js:8437` (build trigger). There is no `POST /api/build/trigger` either — builds also flow through the chat WS path via `handleChatMessage` / `routeToHandler`.

The proper API surface would be:

| Method/Path | Body | Calls |
|-------------|------|-------|
| `POST /api/deploy` | `{ env: 'staging'\|'production' }` | `runDeploy(ws_or_mock, env)` |
| `POST /api/rebuild` | `{}` (uses active TAG) | `triggerSiteBuild(null, readSpec())` |
| `POST /api/restyle` (optional) | `{ instruction }` | `handleChatMessage(buildWs, instruction, 'restyle', spec)` — but this is plan-gated so probably keeps chat semantics |

Each endpoint returns 200 with `{ ok: true, dispatched: true, …details }` on successful dispatch (note: dispatch ≠ completion — for async jobs the body should include a job id and the WS still streams progress), or 4xx with `{ ok: false, reason: 'cli_missing'\|'no_active_site'\|'deploy_in_progress'\|… }` on preflight failure. **The 200/4xx distinction is the architectural contract that the chat path lacks.**

### Estimated scope of fix

| Element | LOC |
|---------|-----|
| `POST /api/deploy` route in `server.js` near `:1085`, calls `runDeploy` with a broadcasting WS mock | ~25 LOC |
| `POST /api/rebuild` route, calls `triggerSiteBuild` | ~15 LOC |
| Update `deployToStaging` / `deployToProduction` / `rebuildSite` in `public/index.html` to `fetch('/api/deploy', {method:'POST', body: …})`, with `addMessage('user', …)` retained for chat continuity | ~30 LOC |
| Optional: keep the chat handlers (`case 'deploy'`, `case 'build'`) as user-typed-natural-language fallbacks, but they are no longer the only path | 0 (status quo) |

**Total ~70 LOC.** Mostly mechanical.

### Hotfix vs deferral

**Ship `POST /api/deploy` + button update in P0.6 hotfix.** Smallest scope (~55 LOC of the 70), biggest unblock — turns the deploy button into a transactional affordance that returns a structured 4xx when Netlify is missing instead of vanishing into chat. Clears the V3-final "operator can ship a site" criterion.

**Defer `POST /api/rebuild` to V2.** Rebuild via Shay Desk's `confirm_build_request` already works for most cases; the Rebuild button hitting the chat is annoying but not blocking. Bundle with the broader chat-vs-API split.

**Skip `POST /api/restyle`.** Restyle is plan-gated — it intentionally goes through chat for the plan card → user-confirms-execute flow. That's a feature, not a bug.

## The architectural question (per V3-final)

V3-final asked: when a failure cluster surfaces architectural issues, name them.

**The architectural issue here is that the chat WebSocket path is doing two jobs at once:**

1. *Conversational surface* — free-form natural language, brain-routed, plan-gated, classifier-driven. Correctly conversational. Should be lossy/ambiguous because language is.

2. *Operator transactional surface* — buttons, deterministic intents, "click → outcome" expectations. Should be lossless and structured, with explicit success/failure contracts. Should look like API calls, not chat messages.

Today (1) and (2) are conflated. Every button is implemented as "type a phrase into chat" and every direct intent has to fight through the classifier and brain-route gates before reaching its handler. Each of the three findings is a different way that conflation breaks:

- Finding 1 — Shay Desk's brain ad-libs an action because the brain layer thinks it's having a conversation; no transactional contract behind the button-equivalent.
- Finding 2 — the chat path's brain-route + silent preflight return drop a real deploy intent on the floor with no log signal.
- Finding 3 — the buttons themselves are just chat-message synthesizers; the conversational layer is load-bearing for transactional intent.

**The right architectural move is to separate the two surfaces:**

| | Conversational surface | Transactional surface |
|-|------------------------|----------------------|
| Transport | WebSocket `chat` messages (today's `/api/shay-shay` + chat WS) | HTTP `POST /api/<verb>` endpoints |
| Caller | User typing free-form | UI buttons + Shay Desk's deterministic handlers + future MCP / scripting |
| Failure mode | Best-effort, may need clarification | Structured 4xx with `reason` |
| Logs | `[classify]`, `[shay-shay]`, brain calls | `[deploy] starting…`, `[rebuild] dispatched…`, structured per-action |
| Existing analogue | `/api/shay-shay` ✓ | `/api/autonomous-build` ✓ (only one); needs `/api/deploy`, `/api/rebuild`, `/api/content-edit` |

Once that split exists, all three findings collapse into the same fix: **the operator-transactional path is HTTP**, the chat path is conversational, and chat messages that look like commands get *re-dispatched* to the HTTP endpoint via a thin classifier+forward layer (so users typing "deploy to staging" still works, but the typing path goes through the same structured endpoint as the button click).

**Highest-priority architectural fix:** introduce `POST /api/deploy` (Finding 3 fix scope) as the proof-of-concept of the split. The button stops being chat-coupled, the failure surface gets `4xx + reason`, and Finding 2's logging/brain-route gaps are closed by making the relevant code path non-chat-coupled. Finding 1 follows the same pattern when content-edit gets its own endpoint, but defer that to V2 since the V2 work already wants to redo the chat-vs-API contract end-to-end.
