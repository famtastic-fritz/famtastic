# P0.5 Diagnostic — `design_brief.approved` flag not flipping on Shay-driven builds

**Date:** 2026-04-27
**Subject site:** `site-lunas-flower-shop` (built today, post-P0.4)
**Reporter symptom:** spec shows `state: "built"` but `design_brief.approved: false`. Post-build chat edit (`change hero headline`) misroutes to `new_site` and Studio surfaces a Design Brief modal instead of a surgical edit.
**Scope:** Diagnose only. **No fixes.**

---

## TL;DR

P0.4's fix (`a2c0baf`) is **correctly written to disk** — `synthesizeDesignBriefForBuild` flips `approved=true` on both Path A (line 7628) and Path B (line 7641). The 7-site backfill **also persisted correctly** — all seven still read `approved: true`.

**Lunas failed because the live `node server.js` process predates the fix.** Studio (PID 52638) started at 12:57:00 PM; the P0.4 code change was written to `server.js` at 13:03:28 PM and committed at 13:08:23 PM; Lunas was built at ~13:16 PM. Studio was never restarted in between, so the in-memory module graph is the pre-P0.4 version where `synthesizeDesignBriefForBuild` does not set `approved=true`. The on-disk fix is real but un-loaded.

This is the same supervision-drift class of bug that blocked the P0 exit gate run — mechanical "is the fix on disk" check passed, but the running process was stale.

## 3. The fix is on disk. Why didn't it persist?

### The fix is correct on disk

`server.js:7606-7645` shows the post-P0.4 implementation:

- **Line 7628** (Path A — no existing brief): `approved: true, // P0.4: synthesizing-for-build implies approval`
- **Lines 7640-7643** (Path B — brief exists but unflipped):
  ```js
  if (spec.design_brief.approved !== true) {
    spec.design_brief.approved = true;
    writeSpec(spec);
  }
  ```

So a fresh execution of the function *should* set `approved=true`.

### But the live process predates the fix

| Event | Wall clock |
|-------|-----------|
| Studio process (PID 52638) started | **12:57:00 PM** |
| `server.js` modified on disk (P0.4 edit) | 13:03:28 PM (`stat -f '%Sm' server.js`) |
| P0.4 committed to git (`a2c0baf`) | 13:08:23 PM (`git log --format='%ci'`) |
| `site-lunas-flower-shop` created | 13:16:34 PM (`spec.created_at`) |
| Lunas build completed | ~13:20 PM (dist file mtimes) |
| This diagnostic | ~13:56 PM (uptime=3561s on `/api/health`) |

`ps -o lstart= -p 52638` confirms the live node process started 12:57:00 PM. `/api/health` reports `uptime: 3561.5s` (≈59 minutes), consistent with the 12:57 start. Studio has NOT been restarted between the P0.4 disk write and Lunas's build.

`lsof -p 52638` shows file descriptor 16r → `/Users/famtasticfritz/famtastic/site-studio/server.js` (inode 20378475). The descriptor was opened at process start; node read the file once into V8's module cache. Subsequent disk edits do not affect the running module graph.

Studio is **launchctl-supervised but not file-watched**. It does not auto-restart on `server.js` mtime changes. The plist's `KeepAlive` only restarts on crash/exit, not on code changes.

### Did `synthesizeDesignBriefForBuild` even fire for Lunas?

Indirectly verified: `spec.design_brief` exists with seven non-trivial keys (`audience`, `avoid`, `content_priorities`, `goal`, `must_have_sections`, `open_questions`, `tone`, `visual_direction`). That object is constructed only by Path A of the function (`server.js:7613-7629`). So the *pre-P0.4 version* of Path A ran — which builds the brief object but does **not** include `approved: true` (it didn't exist as a field in the object literal in the pre-P0.4 source). Path B's flip-if-not-set guard didn't run because Path A was taken (no existing brief at site creation). End result: brief exists, `approved` is whatever default — and looking at the spec, `approved: false` is present in the object, which means **something** wrote it. Most likely the spec-repair-all `normalizeRequiredFields` step or `writeSpec`'s field defaulting wrote `approved: false` as a baseline. (Confirming exact provenance is a 5-min follow-up but doesn't change the diagnosis: the live code did not set it to `true`.)

## Proposed fix locations

**Primary fix (operational, immediate):**

The running Studio process needs to be restarted so the P0.4 module is loaded. Specifically:

```bash
launchctl stop com.famtastic.studio   # KeepAlive will respawn it
# Or, if a Lunas-style site needs immediate recovery without a fresh build:
node -e "const fs=require('fs'); const p='sites/site-lunas-flower-shop/spec.json'; \
         const s=JSON.parse(fs.readFileSync(p)); s.design_brief.approved=true; \
         fs.writeFileSync(p, JSON.stringify(s,null,2));"
```

This is not a code change. It just lands the existing P0.4 fix into the running process.

**Secondary fix (architectural, to prevent recurrence):**

The class of bug here — *fix is committed, fix is on disk, fix is not loaded* — recurred between the P0 exit gate run and the P0.4 verification. Two viable mitigations, ordered by cost:

1. **Auto-restart on `server.js` change.** Add an `fs.watch` (or chokidar) on `server.js` and `lib/*.js` that calls `process.exit(0)` on change. The launchd `KeepAlive` already respawns Studio in <2s with the new ThrottleInterval=30 from P0.3. Cost: ~20 LOC; risk: edits during a live build trigger restart mid-run. Mitigate by gating on `!isBuilding`. Suggested location: `server.js` near the existing process-lifecycle block (search for `process.on('SIGTERM'`, add the watcher next to it).

2. **`/api/health` includes the commit SHA.** Bake `STUDIO_COMMIT=$(git rev-parse HEAD)` into the launchctl plist's `EnvironmentVariables` block; surface it in `/api/health` as `commit: process.env.STUDIO_COMMIT`. The smoke suite's Step 1 then asserts `health.commit === git rev-parse HEAD`. This catches the staleness BEFORE any other smoke step runs. Cost: 3 LOC + plist edit.

The combination of (1) and (2) closes the recurrence pattern: (1) makes the right thing happen automatically, (2) catches the wrong thing happening loudly.

**Adjacent finding (logged, not in P0.5 scope):**

`spec.design_brief.approved` defaults to `false` somewhere — something writes it as `false` rather than leaving it undefined. Worth tracing because the original P0.4 backfill rule keyed on `approved !== true`, which catches both `false` and `undefined`. But for Lunas-class sites in the future, `state === 'built' && interview_completed === true && approved !== true` is what should auto-flip, and either a runtime invariant check (in the spec writer) or a smoke-step assertion would have caught Lunas immediately. P0.4 already added `scripts/smoke-assert-built-spec` for exactly this — running it post-Lunas would have failed:

```bash
./scripts/smoke-assert-built-spec site-lunas-flower-shop
```

That script wasn't run as part of the Lunas build pipeline. Wiring it into the post-build hook (or to the `BUILD_COMPLETED` event handler in server.js) is the third layer of defense — and is closer to the original intent of P0.4's "Step 5b" smoke addition.
